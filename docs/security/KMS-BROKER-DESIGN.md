# In-Container KMS Broker (design note)

This is a design note for letting **tenant workloads inside a
container** perform envelope encrypt/decrypt against the
platform's KMS — *without* ever handing those workloads the
KEK or the cloud KMS credentials. Documentation lands ahead of
implementation; treat the code references here as the
intended shape, not a description of what exists today.

It builds directly on two things already in the tree:

- The KMS envelope layer ([KMS-ENVELOPE-DESIGN.md](KMS-ENVELOPE-DESIGN.md)) —
  `secrets.KMSClient` with `Wrap`/`Unwrap`, backed by `inproc`,
  `vault`, `gcp`, and `aws`.
- The generalized `EnvFile` delivery mechanism
  (`pkg/core/container/envfile.go`, extracted from the OTel
  rollout in #370 and reused for tenant secrets in #491).

## Motivation

Some tenant applications want to encrypt their *own* data
(database fields, blobs, message payloads) under a
platform-managed key, rather than just consuming
platform-delivered secrets. The natural request is "ship us a
KMS SDK like you ship the OTel SDK."

The naive version of that — a client library inside the
container that talks **directly** to Vault / GCP KMS / AWS KMS
— is rejected here. See Non-goals.

## Non-goals / what this is NOT

**Not a direct-to-KMS client in tenant containers.** Doing so
would require distributing KMS credentials (a Vault token, GCP
/ AWS access creds) into every tenant container. That:

- spreads KEK-access credentials into N untrusted workloads —
  a compromised container could wrap/unwrap arbitrary DEKs,
  exhaust KMS quota, or pivot into the cloud account;
- inverts the trust boundary the envelope design exists to
  protect ("the KEK never leaves the trusted daemon");
- cuts against the deny-by-default network isolation and
  per-tenant policy direction.

**Not a new discovery mechanism.** Endpoint discovery reuses
the existing env-stamp + `env_file` substrate. No service
registry, no DNS, no lookup protocol.

## Trust boundary

```
tenant app (compose service)
  │  reads CONTAINARIUM_KMS_ENDPOINT, sends wrap/unwrap requests
  ▼
unix:/run/containarium/kms.sock         ← fixed path inside the LXC
  │  (per-container host socket, bind-mounted in)
  ▼
daemon KMS broker ──► secrets.KMSClient.Wrap/Unwrap ──► Vault / GCP / AWS
   (per-tenant authz here)                    (KEK + KMS creds stay here)
```

The socket carries wrap/unwrap **requests and DEKs**, never the
KEK or KMS credentials. The container holds a plaintext DEK only
transiently (the same property the daemon already maintains in
`Store.encryptForStorage`, which zeroes DEKs after use).

## Discovery and delivery

Mirrors how OTel and tenant secrets are already delivered —
the daemon *stamps* the location in; the container does not
look it up. Two channels, because a nested `docker compose`
service does **not** inherit the LXC's environment:

1. **Incus config-env** (`Manager.SetEnv → environment.CONTAINARIUM_KMS_ENDPOINT`)
   for processes running directly in the LXC. Same path OTel
   uses for `OTEL_EXPORTER_OTLP_ENDPOINT`.
2. **A dotenv file** at a fixed path, consumed by compose via
   `env_file:`. A third `EnvFile` consumer alongside
   `SecretsEnvFile` and the OTel `otel.env`:

```go
// pkg/core/container/kms_envfile.go
const KMSEnvFilePath = "/run/containarium/kms.env"

var KMSEnvFile = EnvFile{
    Path:   KMSEnvFilePath,
    Mode:   "0400", // root-only, like secrets.env
    Header: "# Generated by Containarium KMS broker (delivery=compose). Do not edit.\n" +
            "# docker-compose apps: reference via `env_file: " + KMSEnvFilePath + "`.\n",
}
```

Delivery uses the shared plumbing unchanged:

```go
env := map[string]string{"CONTAINARIUM_KMS_ENDPOINT": "unix:/run/containarium/kms.sock"}
_ = m.WriteEnvFile(containerName, container.KMSEnvFile, env) // same call secrets uses
```

The **fixed in-container path is the contract.** The SDK
defaults to it; the daemon only guarantees the socket and the
env are present.

## Transport: per-container unix socket

The daemon serves a **per-container** unix socket on the host
and bind-mounts it into the LXC with an incus `disk` device
(the same `incus config device add` mechanism used in
`internal/security/scanner.go` and `internal/pentest/external_trivy.go`):

```
host: daemon listens on /var/run/containarium/kms/<container>.d/kms.sock
incus config device add <container> kms-broker disk \
    source=/var/run/containarium/kms/<container>.d \
    path=/run/containarium
# → socket appears inside the LXC at /run/containarium/kms.sock
```

A host-local socket is preferred over a network endpoint
because it needs no network policy hole, can't be reached from
other tenants, and yields caller identity for free (below).

## Identity and authorization

Authorization is enforced **in the daemon**, never in the
container. Identity is the **provenance of the socket**: the
daemon knows which container `/var/run/containarium/kms/<container>.d/kms.sock`
belongs to, so every request on that socket is attributable to
that container's tenant. DEK operations are scoped to that
tenant — a request can only wrap/unwrap keys the tenant owns.

No shared secret travels into the container, so there is
nothing to leak. (If a future network transport is ever added,
it must carry a per-container bearer token, stamped the same
way OTel stamps `OTEL_EXPORTER_OTLP_HEADERS=Authorization=Bearer`,
mapped daemon-side to the tenant.)

## Wire protocol

Newline-delimited JSON over the unix socket — small,
language-agnostic, trivially testable with a fake socket:

```
→ {"op":"wrap","dek_b64":"..."}
← {"wrapped_b64":"...","kek_id":"aws:us-west-2|alias/..."}

→ {"op":"unwrap","wrapped_b64":"...","kek_id":"aws:..."}
← {"dek_b64":"..."}

← {"error":"..."}   // on failure, non-empty error field
```

The broker is a thin adapter over the existing `KMSClient`:

```go
// internal/server/kms_broker.go (sketch)
type kmsBroker struct {
    kms       secrets.KMSClient        // the Vault/GCP/AWS backend, unchanged
    tenantFor func(container string) string
}

func (b *kmsBroker) handle(container string, req kmsReq) kmsResp {
    switch req.Op {
    case "wrap":
        wrapped, kekID, err := b.kms.Wrap(ctx, dek)
        // ... scope/record under tenantFor(container)
    case "unwrap":
        dek, err := b.kms.Unwrap(ctx, wrapped, req.KEKID)
    }
}
```

## CLI surface (CLI-first)

Per the repo convention, the capability lands as a cobra verb
first; the MCP tool is a thin wrapper over the same Go
function.

```
containarium kms-broker enable  <container>   # or a --kms-broker flag on `create`
containarium kms-broker disable <container>
containarium kms-broker status  <container>
```

`enable` (mirroring the `--monitoring` create path):

1. create the per-container host socket dir + start the broker
   listener for it;
2. `incus config device add` the disk device;
3. `WriteEnvFile(KMSEnvFile, …)` + `SetEnv` the endpoint;
4. (optionally) auto-wire the tenant's compose file.

`disable` reverses all four (remove device, remove env file,
stop listener), best-effort like `RemoveOTelEnvFile`.

## Compose wiring

What a tenant's compose file needs — the platform should
generate this automatically (it already does the `env_file`
half for secrets):

```yaml
services:
  app:
    env_file: /run/containarium/kms.env
    volumes:
      - /run/containarium/kms.sock:/run/containarium/kms.sock
```

If the platform does not auto-wire it, "my compose service
can't reach the KMS" becomes the top support ticket — exactly
the failure mode #491 fixed for secrets.

## Thin client SDK

The SDK is "one specific consumer" (like MCP). It carries no
credentials and performs no discovery — it reads one env var
and connects to a fixed socket, the same ergonomics as an OTel
SDK reading `OTEL_EXPORTER_OTLP_ENDPOINT`.

```python
# containarium_kms — thin client (~30 lines)
import os, socket, json, base64

_EP = os.environ.get("CONTAINARIUM_KMS_ENDPOINT", "unix:/run/containarium/kms.sock")

def _call(req):
    s = socket.socket(socket.AF_UNIX); s.connect(_EP.removeprefix("unix:"))
    s.sendall((json.dumps(req) + "\n").encode())
    return json.loads(s.makefile().readline())

def wrap(dek: bytes):
    r = _call({"op": "wrap", "dek_b64": base64.b64encode(dek).decode()})
    if r.get("error"): raise RuntimeError(r["error"])
    return base64.b64decode(r["wrapped_b64"]), r["kek_id"]

def unwrap(wrapped: bytes, kek_id: str) -> bytes:
    r = _call({"op": "unwrap",
               "wrapped_b64": base64.b64encode(wrapped).decode(),
               "kek_id": kek_id})
    if r.get("error"): raise RuntimeError(r["error"])
    return base64.b64decode(r["dek_b64"])
```

A typical application pattern is full envelope encryption of its
own data: generate a DEK, encrypt the payload locally under it
(AES-GCM), `wrap(dek)` to store the wrapped DEK beside the
ciphertext, and `unwrap(...)` to read it back. The platform
holds the KEK; the app holds only ciphertext and a wrapped DEK
at rest.

## Security considerations

- **No KMS creds in containers.** The only thing delivered is a
  socket path. Compromise of a tenant container yields the
  ability to wrap/unwrap *that tenant's* DEKs through the broker
  — bounded by daemon-side authz — and nothing about the KEK or
  the cloud account.
- **Quota / abuse.** The broker is a natural rate-limit and
  audit point; per-container request accounting lives here, not
  reachable by bypassing it (there is no direct KMS path).
- **Socket lifetime.** The socket lives under `/run` (tmpfs),
  evicted on container stop, like `secrets.env`.
- **Blast radius vs. the rejected design.** Direct-to-KMS would
  make one container compromise a cloud-account compromise; the
  broker keeps it a single-tenant, daemon-mediated event.

## Build phases

1. **Broker + `KMSClient` reuse** — daemon-side listener and
   wire protocol, no tenant surface yet. Fully testable with a
   fake socket client and the existing `inproc` backend.
2. **`containarium kms-broker enable/disable/status`** — disk
   device + `WriteEnvFile`/`SetEnv`, mirroring the `--monitoring`
   path; MCP wrapper over the same function.
3. **Auto-wire compose** — reuse the #491 env_file generator;
   add the socket volume mount.
4. **Thin Python client** — published only once 1–3 exist and a
   concrete tenant asks. Keep it demand-driven.

## Open questions

- **DEK ownership model.** Does the broker mint/track DEKs per
  tenant (so `unwrap` can verify the wrapped DEK belongs to the
  caller), or is it a stateless wrap/unwrap oracle scoped only
  by `kek_id` prefix? The former is safer but needs a small
  per-tenant DEK registry.
- **Non-compose, non-Python consumers.** The wire protocol is
  language-agnostic; a Go/Rust/Node client is the same ~30
  lines. Do we ship more than Python at v1? (Default: no.)
- **Auto-wiring opt-in.** Should `--kms-broker` rewrite the
  tenant's compose file, or only emit the snippet for them to
  paste? (Default: emit + document, like the OTel sidecar
  guidance, until auto-rewrite is proven safe.)
